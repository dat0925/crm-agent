# AIエージェントCRM君（仮称）機能仕様書

**対象プロダクト**: サイトマイスター（SiteMeister）
**起案**: 経営トップ指示（修正回数プランのヒアリングを起点とした追加企画）
**作成**: 製品企画部
**version**: 0.2（データドリブン改訂版／夕会用ドラフト）
**前提確認済み事項**
- 連携チャネルは LINE のみ（既存LINE更新依頼エージェントと同一窓口）
- 収益化方針：月2回プランの標準搭載機能として差別化要素に位置づける（追加課金オプションにはしない）
- 各システム（GA4・自社予約システム・Googleビジネスプロフィール 等）からのデータ連携を前提とする

### v0.1 → v0.2 の主な変更点
| 項目 | v0.1 | v0.2（本書） |
|---|---|---|
| エージェントの役割 | 「更新したい箇所はありませんか？」と毎サイクル伺う受動寄りのCRM | 連携データを起点に「今やるべき施策」を先回りで提案する**データドリブンなマーケアドバイザー** |
| 投げかけ内容 | 汎用の更新案内（自然言語不要のpostback3択） | マーケサマリー＋気づき（インサイト）＋優先施策の提案 |
| 「来月に回したい（繰越）」分岐 | あり（要検討事項） | **廃止**（提案ベースのため繰越概念をなくし、見送り／依頼化／無応答に整理） |
| 実装難易度 | 低（既存フローへのハンドオフ＋管理画面のみ） | 中〜高（複数システムのデータ連携・提案生成が加わる。**段階導入を推奨**：12章参照） |

> 注：v0.1は「実装コストが低い」ことを利点に置いていたが、v0.2はデータ連携と提案生成が加わるため開発規模が大きく異なる。価値は高い一方でコストも増えるため、フェーズ分けの意思決定が必要（12章）。

## 1. 概要

AIエージェントCRM君は、サイトマイスター**月2回プラン**に標準搭載する**能動型・データドリブンなWEBマーケ伴走エージェント**である。

GA4（アクセス解析）、自社予約システム、Googleビジネスプロフィール（GBP：MEO内部対策・検索順位レポート等）など各システムから各クライアントのデータを連携し、AIが毎サイクル次の3点を能動的に届ける。

1. **マーケサマリー** — サイト訪問・マップ表示・予約（または問い合わせ）・検索順位などの最新数値
2. **気づき（インサイト）** — 数値の変化から「今注目すべき点」を抽出（例：流入が伸びているのに該当ページが古い）
3. **優先施策の提案** — データの根拠つきで、今サイクルにやるべき具体施策を提示

顧客が提案を選ぶと、既存の更新依頼フロー（ヒアリング〜サイト修正〜公開）へそのままハンドオフする。人手によるマーケCRM（電話・メールでの状況確認や提案）をAIに置き換えることで、

1. コミュニケーションコストの圧縮（人件費・対応時間の削減）
2. データ起点の提案による更新権利の消化率向上 → 「成果に効く更新」化（解約抑止）
3. 「データで伴走するAI」という体験提供による単価訴求の強化・将来のアップセル材料化

を同時に実現する。

## 2. 背景・課題

| 課題 | 内容 |
|---|---|
| 受動依存 | 現行フローは顧客発信が起点。更新できることを忘れる／言い出しにくい顧客が一定数存在し、月2回プランの価値を実感しきれない |
| 「更新しませんか？」の限界 | 汎用の更新案内は「今回は大丈夫」と返される“丁寧なお断り”になりやすく、更新枠の消化率が上がらない。**顧客が動く理由（データ上の根拠）**を示せていない |
| 人的マーケCRMの限界 | 本来は人が毎月データを見て提案する運用が理想だが、契約数のスケールに比例してコストが増大し、AI自動化の方針（資料 p.4, p.6）と矛盾する |
| プラン差別化の弱さ | 月1回（¥15,000）と月2回（¥20,000）の違いが回数のみで体験差が薄い。データドリブンな提案を2回プランの標準搭載にすることで価格差に説得力を持たせる |

---

## 3. 目的（KGI）

- 人手マーケCRMのAI代替によるコミュニケーションコスト削減
- データ起点の提案による更新枠消化率向上 → 契約継続率向上（解約防止策 p.11 と同様の思想）
- 集客成果（流入・予約・MEO）への貢献を可視化し、単価訴求と将来のアップセル材料化

## 4. 用語定義

| 用語 | 定義 |
|---|---|
| 連携データ | GA4・自社予約システム・GBP等から取得する各クライアントのマーケ指標 |
| 巡回サイクル | プランの更新可能回数に基づく月内の配信タイミング（月2回プランは契約開始日基準で約15日ごと、月内2サイクル） |
| マーケサマリー | 各サイクル冒頭でAIが配信する、当月の主要指標レポート |
| 気づき（インサイト） | 数値変化から抽出した「今注目すべき点」 |
| 施策提案 | データの根拠つきで提示する、当サイクルの推奨アクション（複数・優先度付き） |
| 更新ヒアリング | 既存LINE更新依頼エージェントが行う詳細な要望聞き取り（資料 p.5 ステップ2〜3） |
| CRMステータス | 顧客×サイクルごとの進行状態（後述 7章のテーブル参照） |
| ハンドオフ | CRMエージェントの会話から既存更新依頼フローへ処理を引き継ぐこと |

## 5. データ連携（前提）

本機能の中核は「各システムから連携したデータを根拠に提案する」点にある。

| 連携元 | 取得データ（例） | 用途 |
|---|---|---|
| GA4（アクセス解析） | セッション数・流入チャネル・人気ページ・前月比トレンド | 訪問の増減、伸びているページの特定 |
| 自社予約システム | 予約数・曜日別予約・予約経路・稼働率 | 予約の増減、平日／週末の偏りの検知 |
| Googleビジネスプロフィール（GBP） | 表示回数・検索クエリ・ルート/電話タップ・写真閲覧数・クチコミ・**検索順位（MEO）** | マップ露出、検索順位の変動、写真更新の要否 |
| 検索順位レポート | 主要キーワードの順位推移 | 上昇/下降キーワードの把握 |

### 5.1 連携状況とフォールバック
- 連携は顧客単位で管理（`crm_integrations`、7章）。連携済みのソースのみを根拠に提案する。
- 業種により一部ソースが対象外（例：建設業・学習塾は自社予約システムを使わない）。その場合は予約系指標を「問い合わせ」「資料請求」等に読み替え、未連携ソースは提案根拠から除外する。
- 主要ソース（GBP/GA4）がいずれも未連携の顧客は本機能の対象外とし、管理画面で「未連携」表示の上、既存の人手CS運用へフォールバックする。

## 6. 機能要件

### 6.1 対象・適用条件
- 月2回プラン契約中かつ LINE連携済みの全顧客が対象（自動適用、オプトインなし。顧客からの停止申請があれば管理画面から個別OFF可能）
- 主要データソースが未連携の顧客は対象外（5.1）。

### 6.2 トリガー設計（スケジュール／データ同期）
- 契約開始日を起点に、月2回プランは概ね15日ごとに1サイクル（月内2回）
- 日次バッチが「本日が配信対象日」の顧客を抽出
- 配信前に当該顧客の連携データを同期（GA4/予約/GBPのAPI取得）し、最新スナップショットを生成（`crm_data_snapshots`）
- 当該サイクルで既に依頼化済みの顧客はスキップ（重複配信防止）

### 6.3 マーケサマリー生成・配信
- 同期したデータからマーケサマリー（主要指標）＋気づき＋優先施策を生成し、LINE公式アカウントから配信
- 表現はLINEのリッチメッセージ（カード）＋Quick Reply
- 例（飲食店）：
  > Bar Aotetsu 様、こんにちは😊 今月のWEBマーケ状況をまとめました📊
  > サイト訪問 1,240（+18%）／Googleマップ表示 3,580（+24%）／予約 42件（+12%）
  > 「居酒屋 △△駅」検索順位 8位→5位
  > 特に「コース料理」ページの流入が +42%。掲載内容が3か月前のままで、今が更新の好機です。

  Quick Reply：`おすすめ施策を見る` / `レポートを詳しく` / `今月はこのままでOK`

### 6.4 提案ロジック（データシグナル → 施策）
- データの変化（シグナル）を施策にマッピングし、優先度をつけて最大3件提示する。各提案は「根拠データ」「期待効果」を添える。
- マッピング例：

  | シグナル | 提案施策（例） | 期待効果 |
  |---|---|---|
  | 特定ページの流入急増＋内容が古い | 該当ページの最新化 | 流入の取りこぼし防止・CVR改善 |
  | マップ表示の減少＋写真が長期未更新 | GBPへの写真追加 | マップ露出（MEO）の回復 |
  | 検索順位の上昇キーワード | 該当テーマの特集ページ／LP追加 | 季節・需要の取り込み |
  | 予約の曜日偏り（平日稼働低） | 平日限定クーポンの掲出 | 稼働の平準化 |
  | 流入・順位の低下傾向 | 要因の特定と対策提案＋人手CSへの注意喚起 | 離脱の早期検知 |

- 業種別に指標と提案文脈を出し分ける（飲食＝予約、建設＝問い合わせ、学習塾＝資料請求 等）。
- 実装方式は12章のフェーズに応じて、ルールベース（テンプレ）→ LLM生成へ段階的に高度化する。LLM利用時は根拠データを必ず併記し、ハルシネーションを抑制する。

### 6.5 応答分岐
| 顧客の操作 | システム動作 |
|---|---|
| 施策を選択（提案カードをタップ） | 既存LINE更新依頼エージェントの「更新依頼」フローへハンドオフ（資料 p.5 ②③に合流）。ticketに `source=crm_checkin` と `proposal_id` を記録 |
| レポートを詳しく | 流入内訳・曜日別予約・検索クエリ等の詳細カードを追加表示。再度施策提案へ戻れる |
| 今月はこのままでOK | ステータスを「見送り」で確定。会話終了 |
| 自由入力（相談） | 連携データを根拠にアドバイザーとして応答（例：季節メニューの相談）。合意後はハンドオフ可能 |
| 無応答（3日経過） | 1回のみリマインド再送。再送後も無応答なら「未応答」でクローズし、社内CSへ「要フォロー」通知を発行 |

### 6.6 ハンドオフ仕様
- 施策選択後、既存更新依頼エージェントの意図判定をスキップし、直接「更新依頼」インテントのセッションへ遷移（同一LINEトークルーム内で会話継続）
- 生成ticketに `source = crm_checkin`、`crm_cycle_id`、`proposal_id` を付与し、提案→依頼→公開の効果測定で紐付け可能にする

### 6.7 リマインド・エスカレーション
- 送信頻度上限：1サイクルにつき最大2通（初回＋リマインド1回）。スパム化を避ける
- 2サイクル連続で無応答の顧客は「エンゲージメント低下顧客」として管理画面でハイライトし、人手CS／営業の介入対象とする（完全自動ではなく人とAIの併用設計）

### 6.8 管理画面（CRM管理ダッシュボード）
- KPI：今月のレポート自動配信数（＝人手CRM削減効果）、レポート応答率、提案→依頼化率、要フォロー件数
- 顧客一覧：ステータス、AIの提案内容、主要シグナル（連携データ）、直近配信日
- フィルタ：ステータス別（応答待ち／依頼化／見送り／要フォロー 等）
- 顧客詳細（ドロワー）：連携データの状況（GA4/予約/GBP）、AIが参照した当月の主要指標、AIの提案と根拠、会話ログ
- 手動再送ボタン（個別オーバーライドで即時配信）
- 月次レポート：配信数・応答率・依頼化率・見送り率・未応答率（既存の成長支援レポート p.11 #05 と統合表示できると顧客向け価値訴求にもつながる）

## 7. データモデル（テーブル設計案）

### `crm_integrations`（連携状況）
| カラム | 型 | 説明 |
|---|---|---|
| id | uuid (PK) | |
| customer_id | uuid (FK) | 顧客ID |
| provider | text | `ga4` / `reservation` / `gbp` |
| status | text | `connected` / `not_connected` / `n_a`（対象外） |
| account_ref | text | 連携先の識別子（プロパティID、ロケーションID等） |
| connected_at | timestamptz | |

### `crm_data_snapshots`（サイクルごとの指標スナップショット）
| カラム | 型 | 説明 |
|---|---|---|
| id | uuid (PK) | |
| crm_cycle_id | uuid (FK) | 対象サイクル |
| metrics | jsonb | 取得した指標（訪問・マップ表示・予約/問い合わせ・検索順位 等。前月比含む） |
| fetched_at | timestamptz | データ取得日時 |

### `crm_proposals`（生成された施策提案）
| カラム | 型 | 説明 |
|---|---|---|
| id | uuid (PK) | |
| crm_cycle_id | uuid (FK) | |
| title | text | 施策タイトル |
| signal | text | 根拠となったデータシグナル |
| rationale | text | 提案理由（顧客提示用） |
| priority | int | 優先度（1が最優先） |
| status | text | `offered` / `selected` / `dismissed` |
| ticket_id | uuid (FK, nullable) | ハンドオフ後の更新依頼チケット |

### `crm_cycles`
| カラム | 型 | 説明 |
|---|---|---|
| id | uuid (PK) | |
| customer_id | uuid (FK) | 顧客ID |
| plan_type | text | `monthly_2x` 固定（将来拡張用） |
| cycle_start_date | date | サイクル開始日 |
| cycle_seq | int | 月内の何回目か（1 or 2） |
| status | text | `pending` / `sent` / `reminded` / `responded` / `ticket_created` / `skipped` / `no_response` |
| sent_at / reminded_at / responded_at | timestamptz | 各種日時 |
| response_type | text | `request` / `no_need` / `none` |
| created_at / updated_at | timestamptz | |

### `crm_message_templates`
固定コピー（挨拶・リマインド・定型文言）を管理。提案本文はデータから生成するためテンプレ外。
| カラム | 型 | 説明 |
|---|---|---|
| key | text | `checkin_intro`, `checkin_reminder` 等 |
| body | text | 本文（変数差し込み対応） |
| quick_replies | jsonb | Quick Reply選択肢 |
| locale | text | デフォルト `ja`（多言語拡張用） |

### 既存テーブルへの追加想定
- 更新依頼チケットテーブルに `source`（`customer_initiated` / `crm_checkin`）、`crm_cycle_id`、`proposal_id` を追加

## 8. システム構成・処理フロー

```
[日次Cron: crm-checkin-scheduler]
   ↓ 本日対象の顧客を抽出
[crm-data-sync (Edge Function)]
   ↓ GA4 / 予約システム / GBP からデータ取得 → crm_data_snapshots 生成
[crm-proposal-generate]
   ↓ シグナル抽出 → 施策提案を生成（ルールベース→LLM） → crm_proposals 保存
[crm-checkin-send (Edge Function)]
   ↓ LINE Messaging API でマーケサマリー＋提案を配信（Quick Reply付）
   ↓ crm_cycles.status = 'sent'
[LINE Webhook（既存を拡張）]
   ↓ postback を受信
   ├─ 施策を選択 → 既存「更新依頼」フローへハンドオフ → ticket作成（proposal_id紐付け） → status='ticket_created'
   ├─ レポートを詳しく → 詳細カードを返信
   ├─ 今月はこのままでOK → status='skipped'
   └─ 自由入力相談 → アドバイザー応答（合意後ハンドオフ可）
[crm-checkin-reminder (Cron, 3日後)]
   ↓ 無応答に1回だけ再送
   ↓ 再送後も無応答 → status='no_response' → CS通知（Slack/メール）
```

既存のLINE更新依頼エージェント（資料 p.5）に対する**追加コンポーネント**であり、ヒアリング〜修正〜公開のコア部分は再利用する。新規開発は「データ連携・提案生成・能動配信」「CRM管理ダッシュボード」に限定される（v0.1より範囲は拡大）。

## 10. プラン・営業資料への組み込み

- 月2回プラン（¥20,000）の説明に「AIがアクセス解析・予約状況・MEOのデータをもとに、毎月の改善ポイントを提案します」という標準搭載機能として明記
- デモ営業トーク例：「月2回プランは、AIがデータを見て“今やるべき更新”まで提案します。何を直せばいいか分からない、という悩みごとなくなります」
- 既存の比較資料（p.2 月額サービス内訳、p.7 営業フロー資料）への反映が必要なため、営業・マーケ側への展開タイミングを別途調整する

## 11. KPI

| 指標 | 内容 |
|---|---|
| レポート自動配信数 | 人手マーケCRMの削減効果（時間換算） |
| 応答率 | 配信に応答があったサイクルの割合 |
| 提案→依頼化率 | 提案経由で更新依頼に至ったサイクルの割合（更新枠消化の中核指標） |
| 成果貢献 | 依頼化した施策のその後（流入・予約・検索順位）の改善（連携データで追跡） |
| 未応答率／要フォロー件数 | CS介入が必要になった件数 |
| 継続率比較 | 稼働前後の月2回プラン契約者の継続率 |

## 12. 開発スコープ・フェーズ分解（着手レベル）

## 13. リスク・要検討事項

- **データ連携の権限・認証**：GBPのオーナー権限委任、GA4のアクセス権付与、自社予約システムとの内部結合。顧客側の同意・契約条項の整理が前提
- **LLM提案の品質・ハルシネーション**：根拠データの明示、提案テンプレの土台、社内レビュー導線でリスク低減
- **提案の位置づけ**：あくまで「アドバイス」であり成果を保証するものではない旨を明確化（過度な期待・クレーム防止）
- **未連携顧客のフォールバック**：主要ソース未連携時のCS引継ぎ運用を明文化
- **過剰送信**：1サイクル最大2通＋オプトアウト導線
- **コスト増**：API利用・LLM生成のランニングコスト試算が必要（フェーズで段階管理）

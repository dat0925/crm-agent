# AIエージェントCRM君（仮称）機能仕様書

**対象プロダクト**: サイトマイスター（SiteMeister）
**起案**: 経営トップ指示（修正回数プランのヒアリングを起点とした追加企画）
**作成**: 製品企画部
**version**: 0.1（夕会用ドラフト／開発着手レベル）
**前提確認済み事項**
- 連携チャネルは LINE のみ（既存LINE更新依頼エージェントと同一窓口）
- 収益化方針：月2回プランの標準搭載機能として差別化要素に位置づける（追加課金オプションにはしない）

---

## 1. 概要

AIエージェントCRM君は、サイトマイスター**月2回プラン**に標準搭載する**能動型AI CRMエージェント**である。
これまでの更新依頼フローは「顧客が言い出すまで動かない受動型」だったが、本機能により**AIが毎月のタイミングで顧客へ自動的に「更新したい箇所はありませんか？」と伺いを出し**、回答内容に応じて既存の更新依頼フロー（ヒアリング〜サイト修正〜公開）へそのまま引き継ぐ。

人手によるCRM（電話・メールでの状況確認）をAIに置き換えることで、

1. コミュニケーションコストの圧縮（人件費・対応時間の削減）
2. 更新権利の消化率向上による「使われているプラン」化（解約抑止）
3. 月2回プランの価値訴求強化による単価アップの正当化

を同時に実現する。

---

## 2. 背景・課題

| 課題 | 内容 |
|---|---|
| 受動依存 | 現行フローは顧客発信が起点。更新できることを忘れる／言い出しにくい顧客が一定数存在し、月2回プランの価値を顧客が実感しきれないリスクがある |
| 人的CRMの限界 | 経営トップへのヒアリングで「修正回数」を聞かれた際、本来は人が毎月「修正箇所はありませんか？」と確認する運用が想定されていたが、これは契約数のスケールに比例してコストが増大し、AI自動化の方針（資料 p.4, p.6）と矛盾する |
| プラン差別化の弱さ | 現行の月1回（¥15,000）と月2回（¥20,000）の違いは回数のみで、体験としての差が薄い。能動的CRMを2回プランの標準搭載にすることで、価格差に説得力を持たせる |

---

## 3. 目的（KGI）

- 人手CRMのAI代替によるコミュニケーションコスト削減
- 月2回プランの更新権利消化率向上 → 契約継続率向上（解約防止策 p.11 と同様の思想）
- 「AIが伴走している」という体験提供による単価訴求の強化・将来のアップセル材料化

---

## 4. 用語定義

| 用語 | 定義 |
|---|---|
| 巡回サイクル | プランの更新可能回数に基づく月内の伺いタイミング（月2回プランは契約開始日基準で約15日ごとに1回、月内2サイクル） |
| 伺いメッセージ | AIが各サイクルの開始時に顧客へ送る最初のチェックインメッセージ |
| 更新ヒアリング | 既存LINE更新依頼エージェントが行う詳細な要望聞き取り（資料 p.5 ステップ2〜3） |
| CRMステータス | 顧客×サイクルごとの進行状態（後述 6章のテーブル参照） |
| ハンドオフ | CRMエージェントの会話から既存更新依頼フローへ処理を引き継ぐこと |

## 5. 機能要件

### 5.1 対象・適用条件
- 月2回プラン契約中の全顧客が対象（自動適用、オプトインなし。ただし顧客側からの停止申請があれば管理画面から個別OFF可能とする）
- LINE連携済み顧客のみが対象。LINE未連携の顧客は本機能対象外とし、管理画面上で「LINE未連携」表示の上、既存の人手CS運用にフォールバックする

### 5.2 トリガー設計（スケジュール）
- 契約開始日を起点に、月2回プランは概ね15日ごとに1サイクル（月内2回）
- 日次バッチが「本日が巡回対象日」の顧客を抽出し、送信ジョブをキューイングする
- 当該サイクルで既に更新依頼が完了済みの顧客はスキップ（重複送信防止）

### 5.3 伺いメッセージ送信
- LINE公式アカウントからテンプレート文＋Quick Replyで送信
- 例文：
  > いつもサイトマイスターをご利用いただきありがとうございます😊
  > 今月の更新タイミングです。サイトで変更・追加したい内容はございますか？

  Quick Reply：`更新依頼したい` / `今回はおまかせで大丈夫` / `来月に回したい`

### 5.4 応答分岐
| 顧客の回答 | システム動作 |
|---|---|
| 更新依頼したい | 既存LINE更新依頼エージェントの「更新依頼」フローへそのままハンドオフ（資料 p.5 ②③の処理に合流）。ticketに発生元＝「CRM自動巡回」を記録 |
| 今回はおまかせで大丈夫 | ステータスを「利用なし」で確定。会話終了 |
| 来月に回したい | 当該サイクルの権利を翌サイクルへ繰越フラグを立てる（**繰越運用ルールは要確定**。デフォルト方針はデフォルトでは繰越不可／使い切りとし、繰越は将来の有償オプション化を検討） |
| 無応答（3日経過） | 1回のみリマインド再送。再送後も無応答なら「未応答」で当該サイクルをクローズし、社内CS向けに「要フォロー」通知を発行 |

### 5.5 ハンドオフ仕様
- 「更新依頼したい」received後、既存LINE更新依頼エージェントの意図判定をスキップし、直接「更新依頼」インテントのセッションへ遷移（同一LINEトークルーム内で会話を継続）
- 生成されるticketに `source = crm_checkin`、`crm_cycle_id` を付与し、後続のチケット管理・効果測定で紐付け可能にする

### 5.6 リマインド・エスカレーション
- 送信頻度上限：1サイクルにつき最大2通（初回＋リマインド1回）。スパム化を避ける
- 2サイクル連続で無応答の顧客は「エンゲージメント低下顧客」として管理画面でハイライトし、人手CS／営業の介入対象とする（完全自動ではなく人とAIの併用設計）

### 5.7 管理画面（CRM管理ダッシュボード）
- 顧客一覧：プラン種別、直近巡回日、CRMステータス、直近の回答内容サマリ
- フィルタ：ステータス別（送信済み／応答待ち／依頼化／利用なし／未応答／要フォロー）
- 会話ログ閲覧（LINEトーク内容の確認）
- 手動再送ボタン（個別オーバーライドで即時送信）
- 月次レポート：巡回数、応答率、依頼化率、利用なし率、未応答率（既存の成長支援レポート p.11 #05 と統合表示できると顧客向け価値訴求にもつながる）

## 6. データモデル（テーブル設計案）

### `crm_cycles`
| カラム | 型 | 説明 |
|---|---|---|
| id | uuid (PK) | |
| customer_id | uuid (FK) | 顧客ID |
| plan_type | text | `monthly_2x` 固定（将来 `monthly_1x` 拡張時に利用） |
| cycle_start_date | date | サイクル開始日 |
| cycle_seq | int | 月内の何回目か（1 or 2） |
| status | text | `pending` / `sent` / `reminded` / `responded` / `ticket_created` / `skipped` / `no_response` / `carried_over` |
| sent_at | timestamptz | 初回送信日時 |
| reminded_at | timestamptz | リマインド送信日時 |
| responded_at | timestamptz | 顧客応答日時 |
| response_type | text | `request` / `no_need` / `carry_over` / `none` |
| ticket_id | uuid (FK, nullable) | 更新依頼チケットへの参照 |
| created_at / updated_at | timestamptz | |

### `crm_message_templates`
| カラム | 型 | 説明 |
|---|---|---|
| id | uuid (PK) | |
| key | text | テンプレート識別子（`checkin_initial`, `checkin_reminder` 等） |
| channel | text | `line` 固定 |
| body | text | 本文（変数差し込み対応） |
| quick_replies | jsonb | Quick Reply選択肢の配列 |
| locale | text | デフォルト `ja`（将来多言語拡張用） |

### 既存テーブルへの追加想定
- 更新依頼チケットテーブルに `source`（`customer_initiated` / `crm_checkin`）と `crm_cycle_id` のカラムを追加

## 7. システム構成・処理フロー

```
[日次Cron: crm-checkin-scheduler]
   ↓ 本日対象の顧客を抽出
[crm-checkin-send (Edge Function)]
   ↓ LINE Messaging API で伺いメッセージ送信
   ↓ crm_cycles.status = 'sent'
[LINE Webhook（既存を拡張）]
   ↓ Quick Reply の postback を受信
   ├─ 更新依頼したい → 既存「更新依頼」フローへハンドオフ → ticket作成 → crm_cycles.status='ticket_created'
   ├─ 今回はおまかせで大丈夫 → crm_cycles.status='skipped'
   └─ 来月に回したい → crm_cycles.status='carried_over'
[crm-checkin-reminder (Cron, 3日後)]
   ↓ 無応答の顧客に1回だけ再送
   ↓ 再送後も無応答 → status='no_response' → CS通知（Slack/メール）
```

既存のLINE更新依頼エージェント（資料 p.5）に対する**追加コンポーネント**であり、ヒアリング〜修正〜公開のコア部分は再利用する。新規開発が必要なのは「能動的に話しかける起点」と「CRM管理ダッシュボード」の部分に限定される。

## 9. プラン・営業資料への組み込み

- 月2回プラン（¥20,000）の説明に「AIエージェントCRM君が毎月自動で更新のご要望を伺います」という標準搭載機能として明記
- デモ営業トークスクリプトへの追加例：「月2回プランなら、AIが毎月ご様子を伺いますので、更新のタイミングを忘れる心配がありません」
- 既存の比較資料（p.2 月額サービス内訳、p.7 営業フロー資料）への反映が必要なため、営業・マーケ側への展開タイミングを別途調整する

## 10. KPI

| 指標 | 内容 |
|---|---|
| 更新枠消化率 | 巡回経由で依頼化したサイクルの割合 |
| 応答率 | 伺いメッセージへの応答があったサイクルの割合 |
| 未応答率／要フォロー件数 | CS介入が必要になった件数 |
| 継続率比較 | CRMエージェント稼働後の月2回プラン契約者の継続率（稼働前との比較） |

## 12. リスク・要検討事項

- 「来月に回したい」の繰越運用ルール（契約上の解釈整理が必要。デフォルトは使い切りとし、繰越は将来オプション化を検討）
